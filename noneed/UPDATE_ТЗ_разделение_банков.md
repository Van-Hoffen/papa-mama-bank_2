# 🔐 ОБНОВЛЕНИЕ ТЗ: Разделение прав доступа администраторов

## Новое требование: Двухбанковая модель с разделением ролей

### Описание изменений

**Вместо одного администратора ("Родитель")** система теперь предусматривает **двух независимых администраторов**:

- **Мама-администратор** — видит и управляет ТОЛЬКО счётами **Мама-банка**
- **Папа-администратор** — видит и управляет ТОЛЬКО счётами **Папа-банка**

Каждый администратор работает в своей "половине" системы.

## Таблица прав доступа

| Функция | Мама-администратор | Папа-администратор |
|---------|-------------------|-------------------|
| **Одобрение операций** | ✅ Только Мама-банк | ✅ Только Папа-банк |
| **Просмотр вкладов** | ✅ Только Мама-банк | ✅ Только Папа-банк |
| **Просмотр операций** | ✅ Только Мама-банк | ✅ Только Папа-банк |
| **Просмотр статистики** | ✅ Только Мама-банк | ✅ Только Папа-банк |
| **Управление учётными записями** | ❌ Нет доступа | ❌ Нет доступа |

## Изменения в базе данных

### Таблица users (обновление)

```sql
CREATE TABLE users (
  id INT PRIMARY KEY,
  username TEXT UNIQUE NOT NULL,
  password TEXT NOT NULL,
  name TEXT NOT NULL,
  role TEXT CHECK(role IN ('mama-admin', 'papa-admin', 'child')),
  parent_id INT,  -- для связи ребёнка с админом
  bank TEXT CHECK(bank IN ('mama', 'papa', NULL)),  -- для админов
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (parent_id) REFERENCES users(id)
);
```

**Новое поле:** `bank` (TEXT)
- `NULL` для детей
- `'mama'` для Мама-администратора
- `'papa'` для Папа-администратора

### Изменение роли (role)

**Старый вариант:**
```
'admin' | 'child'
```

**Новый вариант:**
```
'mama-admin' | 'papa-admin' | 'child'
```

## Изменения в API endpoints

### 2.3.1 GET /api/operations/pending

**Старое:**
```bash
GET /api/operations/pending
  Auth: admin only
  Response: [ все операции всех банков ]
```

**Новое:**
```bash
GET /api/operations/pending
  Auth: mama-admin OR papa-admin
  Response: [ операции ТОЛЬКО ТОГ БАНКА, к которому привязан админ ]
  
Пример:
- mama-admin видит: операции Мама-банка
- papa-admin видит: операции Папа-банка
```

### 2.3.2 GET /api/deposits?child_id=ID

**Новое:**
```bash
GET /api/deposits?child_id=ID
  Auth: mama-admin OR papa-admin
  Response: [ вклады ТОЛЬКО того банка, к которому привязан админ ]
  
Проверка: если ребёнок создал вклад в Папа-банк, 
Мама-администратор не может его увидеть
```

### 2.3.3 POST /api/operations/:id/approve

**Новое правило:**
```bash
POST /api/operations/:id/approve
  Auth: mama-admin OR papa-admin
  
Проверка: 
- Получаем операцию
- Проверяем её банк (из вклада)
- Проверяем роль админа
- Если банки не совпадают → Error 403 Forbidden
- Если совпадают → одобрить
```

## Изменения в логике приложения

### Регистрация администратора

**При создании нового администратора:**

```javascript
POST /api/auth/register
  Body: {
    username: "mama_parent",
    password: "password123",
    name: "Мама",
    role: "mama-admin",  // или "papa-admin"
    bank: "mama"         // НОВОЕ ПОЛЕ
  }
```

**Валидация:**
- Если role = "mama-admin", то bank ДОЛЖЕН быть "mama"
- Если role = "papa-admin", то bank ДОЛЖЕН быть "papa"
- Если role = "child", то bank ДОЛЖЕН быть NULL

### Фильтрация в панели администратора

**Backend логика:**

```javascript
// В контроллере operations.js
exports.getPendingOperations = (req, res) => {
  const admin = req.user;  // текущий администратор
  
  // Получаем операции
  const operations = db.prepare(`
    SELECT o.*, d.bank, d.amount, c.name as child_name
    FROM operations o
    JOIN deposits d ON o.deposit_id = d.id
    JOIN users c ON d.child_id = c.id
    WHERE o.status = 'pending' 
    AND d.bank = ?  // ФИЛЬТР по банку админа
    ORDER BY o.requested_at DESC
  `).all(admin.bank);  // <-- пропускаем только свой банк
  
  res.json(operations);
};
```

**React компонент AdminDashboard:**

## Правила доступа (Уровень базы данных)

### При одобрении операции

```javascript
exports.approveOperation = (req, res) => {
  const operationId = req.params.id;
  const admin = req.user;  // текущий администратор
  
  // 1. Получаем операцию и вклад
  const operation = db.prepare(`
    SELECT o.*, d.bank FROM operations o
    JOIN deposits d ON o.deposit_id = d.id
    WHERE o.id = ?
  `).get(operationId);
  
  // 2. ПРОВЕРКА ДОСТУПА
  if (operation.bank !== admin.bank) {
    return res.status(403).json({ 
      error: "Недоступно. Это не ваш банк!" 
    });
  }
  
  // 3. Одобряем операцию
  db.prepare(`
    UPDATE operations SET status = 'approved', approved_by = ?
    WHERE id = ?
  `).run(admin.id, operationId);
  
  res.json({ status: 'approved' });
};
```

## Сценарии использования

### Сценарий 1: Два независимых администратора

```
1. Мама создаёт учётную запись (role: mama-admin, bank: mama)
2. Папа создаёт учётную запись (role: papa-admin, bank: papa)
3. Оба добавляют детей в систему

4. Ребёнок Маша:
   - Создаёт вклад в Мама-банке (500 ₽)
   - Создаёт вклад в Папа-банке (2000 ₽)

5. Мама-администратор:
   - Видит ТОЛЬКО вклад в Мама-банке (500 ₽)
   - Видит ТОЛЬКО операции Мама-банка
   - Не видит вклад Папа-банка (2000 ₽)
   - НЕ МОЖЕТ одобрить операцию Папа-банка

6. Папа-администратор:
   - Видит ТОЛЬКО вклад в Папа-банке (2000 ₽)
   - Видит ТОЛЬКО операции Папа-банка
   - Не видит вклад Мама-банка (500 ₽)
   - НЕ МОЖЕТ одобрить операцию Мама-банка
```

### Сценарий 2: Попытка несанкционированного доступа

```
Мама-администратор попытается одобрить операцию Папа-банка:

Request:
POST /api/operations/123/approve
Auth: Bearer <mama-admin-token>

Response:
403 Forbidden
{
  "error": "Недоступно. Это не ваш банк!",
  "allowed_bank": "papa"
}
```

### Middleware проверки доступа

```javascript
// middleware/authMiddleware.js
const checkBankAccess = (allowedBank) => {
  return (req, res, next) => {
    if (req.user.bank !== allowedBank) {
      return res.status(403).json({
        error: `У вас нет доступа к ${allowedBank}-банку`
      });
    }
    next();
  };
};

// Использование:
router.get('/deposits/:id', 
  authenticate, 
  checkBankAccess('mama'), 
  getDeposit
);
```

## Обновление таблицы требований (раздел 2.5)

### 2.5 Панель администратора (ОБНОВЛЕНО)

**Требование 2.5.0 (НОВОЕ):** Система многобанкового разделения

- Два независимых администратора (Мама и Папа)
- Каждый видит ТОЛЬКО "свой" банк
- Невозможно получить доступ к операциям другого банка даже через API
- На уровне БД проверяется совпадение bank админа и bank вклада

**Требование 2.5.1 (ОБНОВЛЕНО):** Главная панель

- Список ожидающих операций ТОЛЬКО этого банка
- Название панели: "Мама-банк" или "Папа-банк"
- Иконка банка в заголовке
- Фильтр автоматический (не видно других банков)

**Требование 2.5.2 (ОБНОВЛЕНО):** Статистика

- Общее количество детей (с вкладами в ЭТОМ банке)
- Общая сумма инвестирована (в ЭТОМ банке)
- Количество ожидающих операций (ЭТОГО банка)
- Общие проценты заработанные (в ЭТОМ банке)

**Требование 2.5.3 (ОБНОВЛЕНО):** История операций

- Просмотр прошлых операций ТОЛЬКО этого банка
- Фильтрация по статусу, дате, типу (внутри банка)

- role='papa-admin' → bank MUST BE 'papa'

## Тестирование разделения доступа (раздел 8)

### 8.1 Функциональное тестирование (ДОБАВЛЕНО)

- [ ] Мама-администратор может видеть вклады только Мама-банка
- [ ] Папа-администратор может видеть вклады только Папа-банка
- [ ] Мама-администратор НЕ МОЖЕТ одобрить операцию Папа-банка
- [ ] Папа-администратор НЕ МОЖЕТ одобрить операцию Мама-банка
- [ ] API возвращает 403 при попытке несанкционированного доступа
- [ ] Статистика показывает данные ТОЛЬКО своего банка
- [ ] История операций фильтруется по банку

### 8.3 Безопасность (ДОБАВЛЕНО)

- [ ] Невозможно получить список вкладов другого банка через API
- [ ] Невозможно изменить bank в токене
- [ ] Попытка API вызова с чужим ID вклада возвращает 403
- [ ] JWT не содержит информацию, позволяющую обойти проверки

## Миграция данных (если уже есть)

Если у вас уже есть админ с role='admin', нужно выполнить:

```sql
-- Обновить существующих администраторов
UPDATE users 
SET role = 'mama-admin', bank = 'mama'
WHERE id = 1;  -- Мама

UPDATE users 
SET role = 'papa-admin', bank = 'papa'
WHERE id = 2;  -- Папа

-- Обновить детей (банк = NULL)
UPDATE users SET bank = NULL WHERE role = 'child';
```
